# Frontend Implementation Instructions

## Primary Objective
Implement frontend code for an AI mentor product that helps CS students improve code through iterative Code Evolution.

## Stack Defaults
Use these unless explicitly changed by user instruction:
1. React + TypeScript strict
2. Vite
3. Tailwind CSS
4. Redux Toolkit for global state
5. Typed API client layer
6. Vitest + React Testing Library

## Theme Source and Token Rules
1. Canonical theme source is app/frontend/src/styles/theme.css.
2. Use semantic theme utilities mapped from theme tokens, including background, foreground, card, border, primary, muted, accent, destructive, ring, and sidebar.
3. Reuse typography and base rules from the theme base layer.
4. Avoid hardcoded brand colors in component classes when semantic theme utilities are available.
5. If token coverage is missing, use a temporary semantic fallback and note the missing token in roadmap notes.

## Source Structure Contract
Most code should live in feature modules.

Top-level structure inside `src`:

```text
src
|
+-- app
|   +-- routes
|   +-- app.tsx
|   +-- provider.tsx
|   +-- router.tsx
+-- assets
+-- components
+-- config
+-- features
+-- hooks
+-- lib
+-- stores
+-- testing
+-- types
+-- utils
```

### Folder Intent
- `app`: app shell, routing, providers, top-level composition.
- `assets`: global static assets.
- `components`: reusable UI used by multiple features.
- `config`: global configuration and environment handling.
- `features`: domain feature modules (default home for new product code).
- `hooks`: cross-feature shared hooks.
- `lib`: reusable library wrappers and preconfigured helpers.
- `stores`: global Redux Toolkit stores/slices.
- `testing`: shared test utilities and mocks.
- `types`: shared TypeScript types.
- `utils`: shared utility functions.

## Feature Module Blueprint
Use this as the default feature shape:

```text
src/features/<feature-name>
|
+-- api
+-- assets
+-- components
+-- hooks
+-- stores
+-- types
+-- utils
```

Rules:
1. Create only required subfolders.
2. Keep feature logic local unless reused by multiple features.
3. Promote code to shared folders only when reuse is proven.

## Coding Rules
1. Keep components focused and composable.
2. Keep data-fetching logic in feature `api` or shared `lib` wrappers.
3. Keep feature state in feature stores; use global stores only for truly global concerns.
4. Prefer typed interfaces and explicit return types for exported functions.
5. Preserve strict TypeScript and lint cleanliness.

## Implementation Rules
1. Always use Tailwind for styling.
2. Do not use external CSS files for component or page styling.
3. No new standalone CSS files for feature or component styling; global theme/base in app/frontend/src/styles/theme.css is the only default exception.
4. Ensure all components are responsive with a mobile-first approach.
5. Before writing code, list the planned component hierarchy as a short tree.
6. Start implementation only after the hierarchy is listed.

## Roadmap and Progress Rules
Track features in:
- `app/frontend/docs/feature-roadmap.md`

When asked to implement a new feature:
1. Ensure a checkbox item exists (create one if missing).
2. Keep item metadata complete.
3. Update status and notes after implementation progress.

Required feature entry format:

```md
- [ ] FEAT-001: Feature title
  - Status: Planned | In Progress | Blocked | Done
  - Milestone: MVP | V1 | Future
  - Priority: High | Medium | Low
  - Owner: Team | Person
  - Depends on: FEAT-XXX | None
  - Acceptance Criteria: <link or brief checklist>
  - Notes: <short context>
```

## Testing Rules
1. Add unit or integration tests for non-trivial logic and behavior changes.
2. Place feature-specific tests near feature code when practical.
3. Use `testing` utilities for shared mocks/helpers.

## UI and Accessibility Rules
1. Build responsive layouts by default.
2. Ensure keyboard and semantic accessibility for interactive components.
3. Use consistent spacing and typography tokens.
4. Prefer semantic theme utilities over literal color classes where possible.
5. Ensure focus-visible styles use theme ring tokens.

## Decision Guidance
If uncertain where code belongs:
1. Put it in the relevant feature folder first.
2. Move it to shared folders only after clear cross-feature reuse.
3. Document the reason in code comments or roadmap notes when making exceptions.
4. For styling decisions, choose Tailwind utilities first and use the theme token system from app/frontend/src/styles/theme.css.
